# AI 3D 真人形象剧 — 设计方案（初稿）

> 用 AI 在 Apple Silicon 上生成长达 30 分钟的 3D 真人形象剧
> 用户创建的 2D/3D 数字人可作为角色加入剧中

---

## 一、核心理念

### 1.1 什么是「3D 真人形象剧」

```
不是传统 3D 动画（Blender/Maya 渲染管线）
也不是纯 2D 视频（已有的短剧/木偶剧）

而是 2.5D 伪 3D 剧：
  ┌─ 角色：2D 数字人 + 多层深度分离 + 简单 3D 变换
  ├─ 场景：Flux 生成宽幅背景 + 前景层（伪 3D 空间）
  ├─ 镜头：Ken Burns 推拉摇移 + 角色透视缩放
  ├─ 动作：预置动效模板（行走/手势/转身）
  └─ 合成：ffmpeg 多轨道 overlay + 画中画
```

### 1.2 为什么不做真 3D

| 因素 | 真 3D | 2.5D 伪 3D |
|:-----|:-----|:-----------|
| 渲染引擎 | Blender/Unity (数 GB) | ffmpeg + OpenCV (已有) |
| GPU 需求 | NVIDIA RTX 4090 | Apple Silicon GPU |
| 模型加载 | 3DMM/NeRF (需训练) | FLUX + MLX (已有) |
| 1 分钟渲染时间 | ~30 分钟 | ~5 分钟 |
| 用户硬件 | 需要独显 | M 芯片即可 |
| 视觉质量 | 高 | **可接受** |

## 三、角色渲染系统（核心）

### 3.1 从 2D 数字人到 3D 角色

```
现有 2D 数字人                     3D 化扩展
┌──────────────┐           ┌──────────────────┐

│              │           │ 左侧+右侧45°)     │
└──────────────┘           └────────┬─────────┘
                                    ▼
                           ┌──────────────────┐
                           │ 角色分层提取       │
                           │ 1. 人脸 (已有)    │
```

### 3.2 角色 3D 变换算法

核心：不是真 3D 渲染，而是 **2D 图像 + 3D 变换矩阵**

```python
def render_character_at_pose(
    avatar_ref: str,       # 数字人参考图
    avatar_meta: dict,     # 面部关键点
    audio: str,            # 口型同步音频
    position: tuple,       # 场景中的位置 (x, y)
    scale: float,          # 透视缩放
    yaw: float,            # 偏转角 (-30° ~ 30°)
    gesture: str,          # 动作模板
) -> list[ndarray]:
    """
    渲染一帧中的角色：
    1. AvatarAnimateStage 生成口型同步视频
    2. 抠图提取人物（已有 matting 能力）
    3. OpenCV 仿射变换模拟 3D 旋转
    4. 按 position/scale 合成到场景
    """
    # 1. 口型同步
    mouth_video = avatar_animate(ref_path, avatar_meta, audio)
    
    # 2. 逐帧处理
    frames = []
    for frame in mouth_video:
        # 抠图
        person = remove_background(frame)
        # 3D 仿射变换
        M = cv2.getRotationMatrix2D(center, yaw, scale)
        transformed = cv2.warpPerspective(person, M, (w, h))
        frames.append(transformed)
    
    return frames
```

### 3.3 多角色支持

```
场景 JSON 示例：
{
  "scene_id": 1,
  "background": "flux://living_room_modern",
  "characters": [
    { "id": "avatar_A", "position": [200, 300], "scale": 0.4, "action": "speaking" },
    { "id": "avatar_B", "position": [600, 320], "scale": 0.35, "action": "listening" }
  ],
  "audio_script": "A: 你好，欢迎来到我的家。 B: 谢谢，这里真漂亮。",
  "duration_seconds": 15
}
```

每个角色独立运行 AvatarAnimateStage，然后按 Z 序（前后关系）合成到场景中。

## 四、场景渲染系统

### 4.1 三层场景结构

```
┌──────────────────────────────────────┐
│  背景层 (Background)                  │
│  Flux 生成 1920×1080 场景图           │
│  提示词: "modern living room, warm   │
│  lighting, depth of field, 8k"       │
├──────────────────────────────────────┤
│  角色层 (Character)                    │
│  数字人口型动画 + 3D 变换 + 阴影      │
│  按 Z-order 排列                      │
├──────────────────────────────────────┤
│  前景层 (Foreground)                   │
│  家具/树木/装饰等遮挡物                │
│  增加景深感（角色走到物体后面时遮挡）  │
└──────────────────────────────────────┘
```

### 4.2 场景类型模板

| 模板 | 描述 | Flux 风格词 |
|:-----|:-----|:-----------|
| 室内-客厅 | 现代/复古/简约 | `living room, warm light` |
| 室内-办公室 | 商务/科技 | `office, natural light` |
| 室外-街道 | 城市/乡村/古街 | `street view, depth` |
| 室外-自然 | 公园/海边/森林 | `natural landscape` |
| 古装 | 宫殿/庭院/客栈 | `ancient Chinese style` |
| 奇幻 | 科幻/魔法世界 | `fantasy, dramatic` |

### 4.3 镜头运动

```
每个场景默认 6-15 秒，带 Ken Burns 镜头运动：
  ┌─ 起始: 场景全景（交代环境）
  ├─ 中段: 推近到说话的角色
  ├─ 对话: 左右角色切换
  └─ 结尾: 拉远到全景
```

## 五、30 分钟长视频策略

### 5.1 分块渲染

```
不一次性渲染 54000 帧，而是：

30分钟 = 120-180 个场景（每个 10-15 秒）

1. 每个场景独立渲染（可并行）
2. 每个场景约 240-360 帧
3. 逐场景保存 clip.mp4
4. ffmpeg concat 拼接

场景渲染→角色渲染→场景合成→保存clip→下一个场景
```

### 5.2 检查点与续跑

```
复用现有的 CheckpointManager：
  - 每渲染完 5 个场景保存一次 checkpoint
  - 支持断点续跑
  - 已完成的场景跳过（idempotency）
```

### 5.3 渲染时间估算

| 组件 | 每场景耗时 | 180场景总耗时 |
|:-----|:---------:|:------------:|
| Flux 场景生成 | ~30s | 90min |
| TTS 合成 | ~5s | 15min |
| AvatarAnimate (单角色) | ~15s | 45min |
| 多角色渲染 (2人) | ~30s | 90min |
| ffmpeg 合成 | ~5s | 15min |
| **总计** | **~85s** | **~255min (4.2h)** |

> 估算基于 M5 Max 128GB 的基准测试

## 八、技术风险与应对

| 风险 | 概率 | 影响 | 应对方案 |
|:-----|:---:|:----:|:---------|
| Flux 场景一致性不足 | 中 | 中 | 固定 seed + style_preset |
| 长视频渲染太慢 | 高 | 高 | 分块渲染 + 2x 速预览 |
| 角色 3D 效果生硬 | 中 | 中 | 先出 MVP，迭代优化 |
| 内存不足 | 中 | 高 | 单场景逐帧释放 + mx.clear_cache |
| 口型与动作不同步 | 低 | 低 | 音频驱动时间轴对齐 |

## 九、待讨论的问题

1. **分辨率**：最终输出 1080p 还是 720p？（720p 渲染快 2x）
2. **帧率**：24fps 还是 30fps？（24fps 少 20% 帧数）
3. **时长**：先做 5 分钟试水还是直接 30 分钟？
4. **角色个数**：最多支持几个角色同框？（建议最多 3 人）
5. **用户输入**：用户提供剧本还是 AI 自动生成？
6. **2D→3D**：先做 2.5D 还是等 3D 模型重建成熟？
