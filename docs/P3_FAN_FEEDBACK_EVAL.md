# P3.4 粉丝反馈雷达评估

> 评估日期：2026-07-06 ｜ 状态：评估完成，结论=**当前不可落地，硬依赖抖音数据 API；不建投机性脚手架，列出明确解锁路径**

## 一、目标

ROADMAP P3 候选："粉丝反馈雷达：抖音数据 → 反馈下集剧本"——读取已发布剧集的播放/互动/评论数据，分析观众反馈，把洞察喂回下一集的 `story_ingest`/`prompt_expand`，形成"发布→反馈→改进"闭环。

## 二、三段链路与各自阻塞点

| 段 | 功能 | 阻塞点 | 当前可做？ |
|:---|:---|:---|:---|
| ① 数据采集 | 从抖音拉取已发布剧集的播放量/点赞/评论/完播率/分享 | **抖音数据 API**（官方 Open API 需开发者资质+审核+数据受限；爬虫违反 ToS 且 ROADMAP 已记"抖音风控 cookie/验证码"） | ❌ 硬阻塞 |
| ② 反馈分析 | 评论 NLP/情感分析/主题提取 → 结构化洞察 | fusion-mlx LLM 已通（chat completions） | ✅ 可做 |
| ③ 反馈回流 | 洞察注入下集 story_ingest 的 outline 指令 | story_ingest 分层故事结构已落地（P1） | ✅ 可做 |

**关键**：②③都依赖①的输出。①是硬阻塞，故整链不可落地。按 Rule 2（不为不存在的数据源建投机脚手架），**不预建②③**——否则是无输入的空壳。

## 三、为什么不能"先用 mock 数据跑通②③"

- Rule 2：单次使用的投机代码不写；mock 数据驱动的②③在没有真实①前无法验证正确性，是"为错误理由通过的测试"（Rule 9）。
- Rule 4：成功标准=真实反馈改进下集剧本质量；mock 数据无法验证此目标。
- 一旦①解锁，②③的 LLM 分析+story_ingest 注入是标准链路，工作量小，到时再建更准（基于真实数据形态）。

## 四、解锁路径

### 路径 A：抖音官方 Open API（推荐，合规）
- 前提：用户申请抖音开放平台开发者资质 + 创建应用 + 通过审核 + 授权获取已发布视频的互动数据权限。
- 落地：新增 `DouyinFeedbackFetcher`（HTTP 客户端，走 fusion-mlx 同款鉴权规范，token 不进日志/URL）→ 拉数据落本地 → LLM 分析 → 注入 story_ingest。
- 触发条件：用户提供有效 Open API 凭据。

### 路径 B：发布端副作用采集（半自动，需用户配合）
- 思路：P2 的 `PublishStage` 发布后，用户手动导出抖音创作者后台的数据（CSV/截图），放入约定目录；`DouyinFeedbackFetcher` 读本地文件。
- 优点：无需 API 资质，规避风控。
- 缺点：非实时，依赖手动导出。
- 触发条件：用户确认愿意定期导出后台数据。

### 路径 C：纯评论分析（最窄，仍需数据源）
- 仅采集评论（非播放量），用 fusion-mlx LLM 做情感/主题分析。
- 阻塞同①：评论同样需 API 或导出。

## 五、对 fusion-mlx 的要求

**当前无**。反馈分析的 LLM 调用走现有 `/v1/chat/completions`，无需新引擎/新端点。若未来评论量极大需批量情感分析，可评估 fusion-mlx 批量推理优化，但那是性能优化非功能缺口。

## 六、推荐

**暂不落地，路径 A 列为首选解锁方案。** 触发条件=用户提供抖音 Open API 凭据（路径 A）或确认定期导出后台数据（路径 B）。届时再建②③，基于真实数据形态设计，避免投机脚手架。
